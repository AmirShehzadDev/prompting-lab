The project we build

Every technique plugs into one coherent system — not nine disconnected toys.

Nimbus is a fictional SaaS: a cloud file-sync & storage product (think Dropbox). We are building Nimbus Support Agent — the AI that answers customer questions, triages tickets, troubleshoots sync failures, looks up accounts, and grounds its answers in the help center.

As the lessons progress, this one agent gains capabilities:

  • Classifies incoming messages (billing / bug / how-to / cancellation / other) — few-shot.
  • Speaks in a consistent voice with guardrails and a refusal policy — role/system prompting.
  • Troubleshoots step by step — chain-of-thought + self-consistency.
  • Runs a pipeline: classify → route → draft → self-critique — prompt chaining.
  • Emits a validated ticket and calls tools (account/order lookup) — structured output + tool use.
  • Acts autonomously across tools in a loop — ReAct agents.
  • Cites the help center and admits when it doesn't know — retrieval-augmented prompting.
  • Is measured and tuned against a golden set — evaluation & optimization.

Shared assets (reused in every lesson)

  • code/kb/ — a small help-center knowledge base (sync errors, billing, plans, sharing).
  • code/data/accounts.json — a fake account/orders datastore for the tool-use lessons.
  • code/nimbus/llm.py — a reusable client wrapper that bakes in the real constraints: retries + backoff, timeouts, token counting, cost estimation, and graceful failure.
  • code/data/evalset.jsonl — a golden set of Q&As and labels we grow as we go.

How each lesson works

Same outcome, two implementations, with the trade-offs made explicit.

Raw

From scratch with the Gemini SDK

You hold the message list, the loop, the retries, the JSON parsing. Nothing is hidden — you learn what the technique actually is at the API level.

LC

The same thing in LangChain / LangGraph

Then we rebuild it with the framework and ask: what did it save us? What did it hide? When is the abstraction worth it, and when does it get in the way?

⚙ Real constraints, baked in

Every lesson respects production reality: token & cost budgets, latency, rate limits, non-deterministic output, evaluation, and graceful failure. No happy-path-only toys.

Each lesson also tells you, for its technique: when to use it, why it works, its failure modes, and the cost / latency / reliability trade-offs.

Setup (do this once)

You need Python 3.10+ and a Gemini API key.

  1. Get a free key from Google AI Studio.
  2. Install the packages used across the course:
  1. Set your API key as an environment variable (the SDK reads it automatically):
ℹ Why an env var?

Hardcoding keys in source is the most common way people leak them on GitHub. The Gemini SDK reads GEMINI_API_KEY (or GOOGLE_API_KEY) from the environment, so your code never contains the secret. We'll follow this everywhere.

That's it — Lesson 1 starts with a 5-line smoke test to confirm everything works, then builds the shared client wrapper the rest of the course depends on.

Retrieval-augmented prompting (RAG)

Way back in Lesson 1, the model answered a sync question by inventing Nimbus's menu items. Tools (Lesson 6–7) fixed account facts, but the help-center knowledge is still made up. RAG is the fix: find the real help article, put it in the prompt, and make the model answer from it — with a citation, or an honest "I don't know".

1 · What RAG is, and why it beats "just ask"

The model knows a lot about the world and nothing about your docs. RAG bridges that gap at prompt time.

A model can only answer from two sources: what it memorized in training, and what you put in the prompt. Your Nimbus help center wasn't in training, so "just asking" forces the model to guess. Retrieval-Augmented Generation flips that: before answering, you retrieve the most relevant passages from your own documents and paste them into the prompt. The model reads and synthesizes, instead of recalling and inventing.

Question"sync paused?"→ Embedquery → vector→ Retrievetop-k similar chunks→ Augmentpaste into prompt→ Generateanswer + cite

"Retrieval" here means semantic search, not keyword matching: we find passages whose meaning is close to the question, even if they share no words. That's what embeddings give us.

2 · Why it works

Three reasons RAG is the default way to give a model private knowledge.

  • Facts beat memory. An answer grounded in retrieved text is far less likely to hallucinate than one pulled from fuzzy training memory.
  • Fresh & private. Update a help article and the next answer reflects it — no retraining. Your data never has to be baked into a model.
  • Auditable. Because you know which passages you injected, the model can cite them, and you can verify the answer against the source.

The catch: RAG is only as good as its retrieval. If you fetch the wrong passages, the model confidently answers from the wrong context. So most of the engineering is in retrieval quality — which is exactly what the failure-modes section is about.

3 · Embeddings: text as searchable vectors

An embedding turns a piece of text into a list of numbers (a vector) that captures its meaning. Similar meanings land close together; we measure "close" with cosine similarity.

Now load the help center, split each article into paragraph-sized chunks, and embed them once to build a simple in-memory index:

A "chunk" is a unit of retrieval. Paragraphs are a reasonable default: big enough to carry meaning, small enough that a hit is precise. Each chunk keeps an id like sync-errors#0 so the model can cite it.

✦ Embed the corpus once, the query every time

Document embeddings are computed once and reused for every question (in production you'd persist them in a vector database). Only the short query is embedded per request — cheap, since embeddings are billed on input only.

4 · Retrieve & ground: the answer, with a citation

Embed the question, score it against every chunk, take the top matches, and tell the model to answer only from them.

representative output — grounded, with a citation
Sorry about the paused sync! To resume it, open the Nimbus icon in your system tray (Windows) or menu bar (macOS), click the Activity tab, and choose "Resume sync". Note that sync auto-pauses on metered or VPN connections unless you allow it in Settings > Network. If it stays stuck, check Settings > Account for a "storage full" banner. [sync-errors#0]

Compare that to Lesson 1, where the model invented "Resume syncing" from the system tray. Now it gives the real path (Activity tab → Resume sync) because it read the actual article — and it cited [sync-errors#0] so you can verify. That is the whole point of RAG.

5 · The honesty win: "I don't know"

Grounding doesn't just improve right answers — it lets the model correctly refuse when the KB has nothing.

answer("Does Nimbus have an app for smartwatches?")
I don't have information about smartwatch support in our help center, so I don't want to guess. I can escalate this to a human who can confirm — would you like me to do that?

Nothing in the KB matches, so the retrieved passages are off-topic, and the grounding rule kicks in: it declines instead of fabricating a confident answer. An honest "I don't know" is a feature, not a failure — it's what makes a support agent trustworthy.

⚠ Retrieval still runs even when nothing fits

Top-k always returns something — the k least-irrelevant chunks, even if all are irrelevant. The model declining here depends on the grounding instruction and on the chunks being clearly unrelated. A score threshold (ignore hits below a similarity cutoff) makes this more robust; see the failure modes.

6 · Failure modes

RAG moves the hard problem from "generation" to "retrieval". These are where it breaks.

FailureWhat happensMitigation
Bad chunkingChunks too big (noisy, dilute the match) or too small (lose context).Tune chunk size; align to natural units (paragraphs/sections); add slight overlap.
Irrelevant retrievalThe right passage isn't in the top-k, so the model answers from the wrong context.Increase k; better embeddings; hybrid (keyword + vector) search; re-ranking.
Nothing relevant existsTop-k returns weak matches and the model answers anyway.Similarity-score threshold + a firm "say I don't know" rule (§5).
Context overflowToo many/long chunks blow the token budget or bury the answer ("lost in the middle").Cap k; keep chunks small; put the most relevant first.
Hallucinated citationsThe model cites an id it didn't use, or invents content not in the passage.Verify cited ids exist; in high-stakes cases, check the claim against the cited text.
Stale indexYou edit an article but the old embedding is still served.Re-embed on change; version the index.
task_type mismatchEmbedding docs and queries with the wrong/identical task type hurts recall.Use RETRIEVAL_DOCUMENT for the KB and RETRIEVAL_QUERY for questions.

7 · Cost, latency & reliability

  • Cost: embedding the corpus is a one-time cost; per request you pay for one tiny query embedding plus the generation, whose input now includes the retrieved chunks. More chunks = more input tokens — so k is a cost dial. Embeddings are billed on input only and are cheap.
  • Latency: one extra embedding call + a similarity search (microseconds for a small corpus; a vector DB handles millions). Usually a small addition before generation.
  • Reliability: a large drop in hallucination when retrieval is good. The failure mode shifts to "retrieved the wrong thing", which you fix with better chunking, k, and thresholds — and measure (Lesson 9: groundedness).

8 · The same thing in LangChain

A vector store + retriever replaces our manual embed / normalize / cosine code.

ConcernRaw SDKLangChain
Embeddingembed_content + manual L2 normalizeGoogleGenerativeAIEmbeddings (handles it)
Storage + searchYour list + cosine loopInMemoryVectorStore + as_retriever
Swap to a real DBRewrite the searchSwap the vector-store class; rest unchanged
ℹ This is a strong case for the framework

RAG is where LangChain's abstractions pay off: the vector-store interface means you can prototype with InMemoryVectorStore and later swap in Pinecone / pgvector / Chroma by changing one class. It also quietly handled the normalization gotcha we had to do by hand. The cost: less insight into how retrieval actually scores — which is exactly what you most want to debug when recall is bad.

🔨 Try / modify this

  1. See retrieval directly. Print retrieve("how do I share a folder?") and read the chunk ids it returns. Did it find sharing#0?
  2. Break grounding. Remove the RAG_RULES and re-ask the smartwatch question. Watch it start inventing again — proof the rule does the work.
  3. Tune k. Set k=1 then k=6. Too low misses context; too high adds noise and cost. Find the sweet spot for this KB.
  4. Add a threshold. Make retrieve drop chunks with cosine < 0.55. Confirm the smartwatch question now retrieves nothing and the agent cleanly declines.
  5. Edit the KB. Change a fact in kb/plans.md, re-run, and watch the answer update with no retraining. That's RAG's freshness win.

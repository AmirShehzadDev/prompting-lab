Foundations & your first prompt

Before any clever technique, you need the boring-but-critical basics: what an API call is, what tokens cost, why output isn't repeatable, and how to not crash when the network hiccups. We'll build the llm.py wrapper the whole course reuses — then meet our first technique, zero-shot prompting.

1 · The mental model

An LLM API is simpler than it looks. Strip away the hype and it's this:

You send text. The model predicts the text that should come next, one token at a time. You get that continuation back. That's the whole interface.

Three things are worth burning into memory on day one:

  • It's stateless. The model remembers nothing between calls. A "conversation" is an illusion you create by re-sending the whole history every time. This is why long chats get expensive — you pay for the entire transcript on every turn.
  • Messages have roles. Your input is a list of turns, each tagged user or model (Gemini calls the assistant's role model, not "assistant"). There's also a separate system instruction — standing orders that sit above the conversation (Lesson 3 is all about it).
  • It's non-deterministic. Same input can give different output. You tune that with temperature (0 = focused and near-repeatable; higher = more varied/creative).

2 · Your first call (the smoke test)

If this runs, your install and API key work. Five lines.

representative output
Cloud file sync automatically keeps your files updated and identical across multiple devices by storing and managing them in the cloud.

Line by line:

  • genai.Client() — builds the API client. No key in your code; it's read from the environment. (If both GOOGLE_API_KEY and GEMINI_API_KEY are set, GOOGLE_API_KEY wins.)
  • model="gemini-2.5-flash" — which model to call. This is just a string the client passes through. Swapping models is a one-string change — remember that; it matters in a minute.
  • contents="…" — your prompt. A bare string is shorthand for "one user turn with this text."
  • resp.text — the model's reply as a string. (It can be None if the response was blocked or wasn't plain text — we'll handle that in the wrapper.)
⚠ Heads-up on model lifecycles

As of this writing gemini-2.5-flash is live and has stable, published pricing, so we use it — but Google has it on a deprecation schedule (shutdown 2026-10-16). That's exactly why we'll keep the model name in one constant: when it sunsets, you change a single line to the current flash model and every lesson keeps working. Model IDs are disposable; the techniques aren't.

3 · Tokens, cost & latency

Every call spends three budgets: money, time, and a rate-limit allowance. You can't manage what you don't measure.

What's a token?

Models don't see characters or words — they see tokens, chunks of roughly 4 characters (~¾ of a word in English). "Nimbus sync failed" is about 5 tokens. You pay per token, separately for input (your prompt) and output (the reply), and the model has a maximum number of tokens it can handle at once (its context window).

Measuring it

You can count input tokens before you spend money, and read exact usage after every call.

representative output
estimated input tokens: 18 Unfortunately, **you cannot directly share a Nimbus folder with someone who doesn't have a Nimbus account.** Nimbus is designed for collaborative work within its own ecosystem... (answer continues until the 300-token cap) prompt : 18 candidate: 300 total : 318
✗ Easy bug to hit

count_tokens(...) gives you .total_tokens (singular). The post-call usage_metadata gives you .total_token_count (with _count). Different objects, different names. Mixing them up is a classic AttributeError.

Turning tokens into dollars

Pricing is quoted per million tokens, with input cheaper than output. Verified current rates (USD / 1M tokens):

ModelInputOutputGood for
gemini-2.5-flash$0.30$2.50Our default — fast, cheap, capable
gemini-2.5-flash-lite$0.10$0.40High-volume, simple tasks (classification)
gemini-2.5-pro$1.25$10.00Hard reasoning (≤200k-token prompts; pricier above)

That 318-token call above costs about 18/1e6 × $0.30 + 300/1e6 × $2.50 ≈ $0.00076 — under a tenth of a cent. Trivial once. Multiply by 50,000 support tickets a day and model choice and prompt length become real money. That's the mindset.

◆ Two pricing gotchas

1. Output is ~8× input on flash — verbose answers cost more than long prompts. 2. Pro is context-tiered: prompts over 200k tokens jump to $2.50 / $15.00. Embeddings (Lesson 8) are billed on input only. Our cost helper handles the flat case; we'll note the tier when it matters.

Latency: the other budget

Bigger models think longer. Rough rule: flash-lite < flash < pro for latency, and the reverse for capability. Output length dominates wall-clock time (each token is generated sequentially), so max_output_tokens is a latency lever, not just a cost lever. For a live support chat, a 2-second flash answer usually beats a 9-second pro answer that's marginally better.

4 · The four real constraints

Toy scripts ignore these. Production code can't. Each one motivates a piece of the wrapper we build next.

ConstraintWhat goes wrongHow we handle it
Non-determinismSame prompt, different answer; a test that passed yesterday fails today.Set temperature=0 for tasks that should be stable (classification, extraction); accept variance elsewhere and evaluate (Lesson 9).
Rate limitsToo many requests too fast → HTTP 429 and your script dies mid-batch.Catch 429, wait, and retry with exponential backoff.
Transient failuresThe API returns 500/503 for a few seconds for no reason you control.Same retry path as 429; give up after N attempts.
Graceful failureA blocked or empty response makes resp.text be None; .strip() then crashes.Detect empty/blocked output and raise a clear error instead of a mystery NoneType crash.

5 · Build the shared wrapper: nimbus/llm.py

This is the most important file in the course. Every later lesson imports it. It turns the raw SDK into a client that retries, times out, tracks spend, and fails loudly-but-cleanly.

Create a folder nimbus/ and put this in nimbus/llm.py:

representative output of python -m nimbus.llm
Cloud file sync is the automatic synchronization of files across multiple devices via a cloud-based server, ensuring access to the latest versions from anywhere. --- 1 calls | 11 in + 28 out tokens | ~$0.000073
✦ What this buys you

From now on a lesson can just write llm = LLM(); text = llm.generate(prompt) and get retries, a timeout, cost tracking, and clean errors for free — while still being a thin, readable layer over the real SDK. We didn't hide the API; we hardened it.

ℹ Two honest notes

Timeouts vs. retries: a network timeout raises an httpx exception, not an APIError; catch those too if you want timeouts retried. We keep the wrapper focused on the HTTP status codes the API actually returns. Built-in retries: the SDK can retry for you via HttpOptions(retry_options=...) — we hand-roll it here so you can see exponential backoff, which is the thing worth understanding.

6 · Zero-shot prompting

Our first real technique — and the baseline every other technique is measured against.

Zero-shot means: state the task in plain language and ask, giving the model no examples of the task being done. You're relying entirely on what the model already learned during training. It's the simplest possible prompt, and very often it's enough.

Why it works

Modern models are instruction-tuned — fine-tuned to follow natural-language requests. For any task well-represented in their training data ("summarize this", "answer this question", "fix this grammar"), they generalize from the instruction alone. No examples needed.

When to use it

  • The task is common and the output format is loose (a free-text answer).
  • You're prototyping and want a baseline before investing in anything fancier.
  • Token budget is tight — zero-shot is the cheapest prompt there is (no example tokens).

Here's our Nimbus support baseline — a customer question, answered zero-shot through the wrapper:

representative output
It's frustrating when Nimbus pauses syncing! Here's a comprehensive troubleshooting guide, starting with the simplest solutions: **Quick Fixes (Try These First!)** 1. Check Your Internet Connection (open a browser, try a few sites)... 2. Restart Nimbus Desktop App (fully quit from the system tray, then reopen). 3. Restart Your Computer. **Nimbus-Specific Troubleshooting** 4. Check Nimbus Status Page (search "Nimbus Note status page")... 5. Manually Initiate Sync — look for a "Sync" button or option within the ... (answer continues until the 350-token cap) --- 1 calls | 21 in + 350 out tokens | ~$0.000881

Failure modes (look closely at that answer)

It reads great — and that's the trap. Zero-shot has no idea what Nimbus actually is:

  • Confident hallucination. It assumes Nimbus is "Nimbus Note" (a real, different product), invents a "Nimbus Note status page", and guesses menu options like "Manually Initiate Sync" — all plausible-sounding, none of them facts about our Nimbus. The model can't know a product it never saw in training. Lesson 8 (RAG) fixes this by grounding answers in the actual help center.
  • No format control. This run was a long markdown guide; another might be a terse paragraph. Fine for chat, useless if you need to parse the output. Lessons 2 & 6 fix this.
  • No persona or guardrails. Generic tone, no escalation policy, will happily answer off-topic questions. Lesson 3 fixes this.
  • Non-deterministic. We set temperature=0.3, so re-running gives a slightly different answer each time.
✦ The whole course in one idea

Every later technique exists to fix a specific weakness you just saw in this zero-shot baseline. Keep this answer in mind — we'll measure everything against it.

7 · The same thing in LangChain

Now the contrast that runs through the whole course: build it again with the framework, and ask what changed.

LangChain wraps each provider in a uniform ChatModel interface. You build messages from typed objects (SystemMessage, HumanMessage) and call .invoke(). Here's the zero-shot baseline, LangChain-style:

representative output
(a long markdown troubleshooting guide, much like the raw version — e.g. "check your system tray, right-click the Nimbus icon, select Quit, then reopen"...) {'input_tokens': 21, 'output_tokens': 350, 'total_tokens': 371, 'input_token_details': {'cache_read': 0}}

What did the framework do for us?

Honestly, not much yet — and that's the point of starting here. For a single zero-shot call, LangChain mostly adds a layer of typed objects. But notice what it quietly handled:

ConcernRaw SDK (our llm.py)LangChain
Retries / backoffWe wrote the loop ourselves (≈15 lines)Built in via max_retries=
Usage accountingWe read usage_metadata and price itNormalized to resp.usage_metadata['input_tokens'] — same keys across all providers
MessagesA string or types.Content objectsHumanMessage / SystemMessage objects
Swapping providersRewrite against a new SDKSwap the chat-model class; rest of the code is unchanged
What it costs youNothing hidden — you see every callA dependency, an abstraction to learn, and behavior you don't directly control
ℹ The recurring question

For one call, the raw SDK is barely more code and far more transparent. LangChain's value shows up when you compose things — chains, retrievers, agents, swapping models — which is exactly what the later lessons build. We'll keep doing both so you can judge the trade-off yourself, every time, instead of taking it on faith.

8 · Trade-offs: when to reach for zero-shot

Every technique in this course gets this scorecard: cost, latency, reliability, and the one-line verdict.

DimensionZero-shot
Cost💚 Lowest. No example tokens — you pay only for the instruction + question + answer.
Latency💚 Lowest. Shortest prompt, single call.
Reliability🟠 Lowest of the techniques. No format control, no grounding, will hallucinate specifics confidently.
Reach for it when…The task is common, the format is loose, and a wrong-but-plausible answer is acceptable (or you're just establishing a baseline).
Reach for something else when…You need a fixed output shape (→ few-shot / structured output), product-specific facts (→ RAG), a consistent voice (→ system prompt), or multi-step reasoning (→ CoT).
⚠ One outdated trick, noted and dismissed

You'll see old blog posts add things like "You are a world-class expert. I'll tip you $200. Take a deep breath." On today's instruction-tuned models these add tokens and noise for little to no gain. Clear instructions and (when needed) examples beat incantations. We'll keep prompts clean; Lesson 10 has the full myth-busting list.

🛠 Try / modify this

  1. Feel the non-determinism. Run zero_shot.py three times at temperature=0.9, then three times at temperature=0.0. Watch the variance shrink. This is why stable tasks pin temperature to 0.
  2. Make the cost real. Add a loop that calls llm.generate() 20 times, then print llm.usage.report(). Now change DEFAULT_MODEL to "gemini-2.5-flash-lite" and compare the dollar figure. One constant, big swing.
  3. Trigger graceful failure. Set max_output_tokens=1 and ask for a long answer — then inspect what finish_reason you get. Confirm the wrapper raises a clear error instead of crashing on None.
  4. Catch a hallucination. Ask "What's the keyboard shortcut to force-sync in Nimbus?" zero-shot. The model will likely invent one. Note it down — in Lesson 8 we'll ground this same question in the real help center and watch the made-up answer disappear.
  5. Prove the abstraction. In the LangChain version, change model= to "gemini-2.5-flash-lite" and confirm nothing else needs to change. That provider-swap ergonomics is LangChain's core pitch.

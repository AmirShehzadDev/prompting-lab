Structured output & tool use

Every lesson so far ended with us parsing prose — regexing a FINAL: line, normalizing a label, hoping the format held. Now we stop hoping. The model can return schema-validated JSON you get as a typed object, and it can call functions to fetch real account data. This is the leap from chatbot to agent.

1 · The problem with parsing prose

Free text is great for humans and terrible for code.

Look back at what we've been doing to get machine-usable data out of the model:

  • Lesson 2: normalize() lower-cased the label and hoped it matched our set.
  • Lesson 4: parse_final() ran a regex on a FINAL: line and returned None when the format drifted.
  • Lesson 5: the critic's PASS/FAIL was string-matched.

Every one of those is a guess that the model formatted its output exactly right. At scale, some fraction won't — and your pipeline silently mishandles them. What we actually want is a typed object with guaranteed shape: an enum that's always one of the allowed values, a boolean that's always a real boolean. Two features get us there:

FeatureWhat it gives youNimbus use
Structured outputThe model's reply is constrained to a JSON schema; you get a validated object.A support Ticket (intent, severity, needs_human…).
Tool use (function calling)The model can call your Python functions and use the results.Look up a real subscription / order history.

3 · Why it's reliable (not just "asking nicely")

There's a real mechanism here, not a politeness trick.

When you pass response_schema, the API uses constrained decoding: at each step it only lets the model pick tokens that keep the output valid against the schema. The model literally cannot emit "severity": "kinda bad" when the schema says the field is one of four enums — those tokens aren't allowed. That's categorically stronger than putting "please reply in JSON" in a prompt and praying.

Then the SDK parses and validates the result into your Pydantic model before you ever see it. Two layers — constrained generation, then validation — which is why this replaces fragile parsing outright.

✦ This retroactively upgrades earlier lessons

Remember Lesson 4's flaky parse_final() regex and Lesson 2's normalize()? Both become unnecessary with a schema (intent: Literal[...], a typed plan/total). When you need a machine-readable answer, prefer a schema over "ask for a format + parse it". Use prose only when a human is the consumer.

4 · Tool use: letting the model fetch real data

Structured output shapes what comes out. Tool use lets the model pull real facts in — the cure for the hallucinated specifics we saw in Lesson 1.

A "tool" is just a Python function you expose. The model, seeing the function's name, docstring, and typed parameters, can decide to call it; you run it; the result goes back to the model. Here are two tools backed by a tiny account datastore (data/accounts.json):

The docstrings and type hints aren't decoration — the SDK turns them into the schema the model uses to decide when and how to call each tool. Write them as if for the model, because they are.

The simplest path is automatic function calling: pass the functions as tools and the SDK runs the whole call-execute-respond round trip for you.

representative output
Yes — I can see two charges on your account dated 2026-06-01, both for $10.00 (order NB-1001 and NB-1002, the second labelled a duplicate). I can't process refunds myself, but I'll escalate this duplicate charge to a human billing agent to get it reversed. Anything else I can check?

The model recognized it needed order data, called get_recent_orders("alice@example.com"), saw the two charges, and answered with real facts — in the Nimbus voice, still refusing to promise the refund itself. That grounding is what Lesson 1's zero-shot baseline could never do.

5 · Under the hood: the manual tool loop

Automatic calling is convenient, but you'll want manual control for agents (Lesson 7) — to log, validate arguments, or gate dangerous actions. Here's the mechanism it hides.

representative output
model wants: get_subscription {'email': 'carol@example.com'} Carol's Pro plan is currently marked "past_due" (it was due to renew 2026-06-20), so the account isn't in good standing right now. Updating the payment method should restore it — want me to walk you through that or escalate to billing?

Three explicit steps: the model requests a call, you run it (the control point where you'd validate args or require approval), and the model uses the result. Loop that decision repeatedly and you have an agent — which is exactly Lesson 7.

6 · Failure modes & safety

Structured output is robust; tool use opens a door you must guard.

FailureWhat happensMitigation
Over-complex schemaDeeply nested / huge schemas degrade quality and can fail to fill.Keep schemas flat and small; split into multiple calls if needed.
Hallucinated argumentsNo email given, so the model invents one and calls the tool with garbage.Validate args before executing; have the tool return a clear error; make the model ask for missing info.
Wrong / missing tool callModel answers from memory instead of calling the tool (or calls when it shouldn't).Clear tool docstrings; a system-prompt nudge ("use tools for account facts"); evaluate it (L9).
Tool errorsThe function raises; the agent stalls or leaks a stack trace.Catch inside the tool; return {"error": "..."} so the model can recover gracefully.
parsed is NoneRare, but a blocked/empty response yields no object.Our generate_json raises clearly; retry or fall back.
✗ Tools are actions — treat them as a security surface

A tool can read data, spend money, or change state. The model decides when to call it, and that decision can be steered by malicious user text (prompt injection from Lesson 3). Rules: validate every argument, grant least privilege (read-only where possible), gate destructive or irreversible actions behind explicit human approval, and never treat the model's choice to call a tool as if a human authorized it. The manual loop (section 5) exists precisely so you have that control point.

7 · Cost, latency & reliability

  • Structured output: essentially the same cost/latency as a normal call — you're just constraining the tokens. Pure win for machine-readable results.
  • Tool use: each tool call is an extra round trip (model → tool → model), so a question needing two lookups is ~3 model calls. Latency and cost scale with the number of tool steps — the price of grounding in real data.
  • Reliability: a large jump. Schemas remove parsing failures; tools remove hallucinated facts. The new risk isn't format, it's actions — which is why section 6's safety rules matter.

8 · The same thing in LangChain

Two one-liners: with_structured_output and bind_tools.

ConcernRaw SDKLangChain
Structured outputresponse_schema= → resp.parsedllm.with_structured_output(Ticket)
Toolstools=[fn]llm.bind_tools([fn])
Executing toolsAutomatic, or manual loopbind_tools returns .tool_calls; you execute (or use create_agent, L7)
ℹ The key difference

Raw Gemini's automatic function calling runs the whole loop for you. LangChain's bind_tools deliberately stops at "here's the call I want to make" and hands it back — you run it. To get a full autonomous loop in LangChain, you graduate to create_agent / LangGraph, which is Lesson 7.

9 · Trade-offs & try / modify

TechniqueCostReliabilityReach for it when…
Structured output💚 ~same as a normal call💚💚 Guaranteed shapeAny machine-readable result (tickets, labels, extraction).
Tool use🟠 Extra round trip per call💚 Real facts, no hallucinationThe answer depends on live data or an action.

🛠 Try / modify this

  1. Trust the enum. Run build_ticket() on ten wild messages and confirm severity is always one of the four values — no normalization needed.
  2. Replace old parsing. Rewrite Lesson 4's plan/cost answer with a Plan(plan: Literal[...], total: float) schema and delete parse_final.
  3. Force a hallucinated arg. Ask "was I charged twice?" with no email. Does the model invent one, or ask for it? Add a system-prompt rule to make it ask.
  4. Add a tool. Write get_plan_limits(plan) and ask a question that needs both it and get_subscription — watch the model chain two tool calls.
  5. Gate an action. Add a fake cancel_subscription(email) tool, then use the manual loop to require a typed "yes" before executing. That's the safety pattern for L7.
✦ What you built

A schema-validated Ticket (no more regex), two account tools grounded in real data, and both the automatic and manual function-calling loops — raw and in LangChain. The manual loop is the seed of the agent.

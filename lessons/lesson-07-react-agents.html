ReAct-style agents

Lesson 5's pipeline ran fixed stages in a fixed order. But some requests need the system to decide for itself what to do next — look up the order, then check the plan, then answer. That's an agent: a loop where the model reasons, calls a tool, observes the result, and repeats until it's done. Powerful, and the easiest thing in this course to let run away.

1 · Pipeline vs. agent: who decides the steps?

The difference is who controls the control flow.

In Lesson 5 you wrote the steps: classify, then route, then draft, then critique. The order was fixed in Python. That's perfect when the steps are always the same. But consider:

"Hi, I'm alice@example.com. I think I got double-charged and I want to know what plan I'm on so I can decide whether to downgrade."

Answering that needs a look at the orders AND a look at the subscription — but a different question might need neither, or one, or three lookups. You don't know the steps in advance. So instead of hard-coding them, you let the model choose: give it tools and let it loop until it has what it needs. That loop is the agent, and the price of that flexibility is unpredictability you must bound.

Pipeline (L5)Agent (this lesson)
Who picks the stepsYou, in codeThe model, at runtime
Control flowFixed, linearDynamic loop
Predictable costYesNo (bound it!)
Best forKnown, repeatable tasksOpen-ended, multi-step requests

2 · ReAct: reason, act, observe

The pattern that powers agents is a loop of three moves, repeated until the model can answer.

Reasonwhat do I need? → Actcall a tool → Observeread result ↻ repeat → Answer

ReAct ("Reasoning + Acting") interleaves thinking and tool use: the model reasons about what it needs, takes an action (a tool call), observes the result, and loops — accumulating information until it can respond. It's Lesson 4's reasoning plus Lesson 6's tools, in a cycle.

Classic ReAct vs. how it works today

The original technique made the model emit text scaffolding you parsed by hand:

Modern tool-calling APIs make this native: instead of text you parse, the model emits a structured function_call (exactly the mechanism from Lesson 6), and you reply with a structured function_response. Same loop, no fragile parsing. We build the native version — it's what every production agent uses now.

ℹ "Thinking" models reason inside each step

On a Gemini 2.5 thinking model, the "Reason" part can happen internally (Lesson 4) before the model decides which tool to call. So a modern agent step is: hidden reasoning → emit a tool call (or a final answer). The loop structure is identical.

3 · Build the raw think/act/observe loop

It's the Lesson 6 manual tool call, wrapped in a for loop with two hard guards. Reusing those tools, the whole agent is ~20 lines.

representative run
step 1: get_recent_orders({'email': 'alice@example.com'}) step 1: get_subscription({'email': 'alice@example.com'}) [2 steps, 1369 tokens] It looks like you were double-charged on 2026-06-01 for your Plus monthly plan. I've also confirmed that you are currently on the Plus plan. To resolve the double charge, please contact our billing support, and we'll be happy to assist you with a refund.

The model decided, on its own, to call both tools in a single step (a parallel tool call, hence two step 1 lines), then answered on the next — finishing in 2 steps, using real data. We never told it which tools or what order; it reasoned its way there. That is the agent.

4 · The guards that keep it from running away

An agent is a loop whose exit condition is "the model decides it's done." If the model never decides, the loop never ends. Guards are not optional.

Our 20-line agent already has the two that matter most:

  • Step cap (max_steps) — a hard ceiling on iterations. Without it, a confused model can call tools forever. This is the single most important guard.
  • Token / cost budget (token_budget) — stop if the accumulated tokens cross a line, so a stuck agent can't run up a bill.

In production you'd add more: a timeout (wall-clock), dedup of identical repeated tool calls (a classic loop symptom), per-tool argument validation and human approval for any destructive action (Lesson 6 section 6), and logging of every step for audit. The rule: an agent must be unable to do unbounded work or irreversible harm.

⚠ The frameworks ship this guard too

LangGraph's agents have a recursion_limit for exactly this reason — it raises rather than looping forever. Whatever you build on, find the loop ceiling and set it deliberately. A missing step cap is the most common way an agent demo becomes a billing incident.

5 · Failure modes

Agents fail in more, and weirder, ways than anything earlier — because they choose their own path.

FailureWhat happensMitigation
Infinite / oscillating loopCalls the same tool repeatedly, or ping-pongs between two, never answering.max_steps cap; dedup repeated calls; clearer tool descriptions.
Cost blowupEach step is a full model call; a 10-step detour is 10× the cost.Token budget; cheap model; cap steps; prefer a pipeline when steps are known.
Wrong / hallucinated tool argsInvents an email, or calls the wrong tool for the job.Validate args in the tool; return {"error": ...}; make the agent ask for missing info.
Never finishesKeeps "gathering more" instead of answering.System-prompt nudge ("answer once you have enough"); step cap forces a stop.
Over-actingCalls tools for a question it could answer directly — wasted calls.Tell it tools are for account facts only; evaluate tool-use rate (L9).

6 · Cost, latency & reliability

  • Cost & latency: the least predictable in the course. A request might take 1 step or hit your cap of 6 — each step a full model round trip plus a tool call. Budget for the worst case, not the average.
  • Reliability: agents handle open-ended requests no fixed pipeline could — but the flexibility that makes them powerful also makes them harder to test and bound. Treat the step cap and budget as correctness requirements, not nice-to-haves.
  • Lever: a tight system prompt ("use tools only for account facts; answer as soon as you can") measurably reduces wasted steps. Behavior shaping (Lesson 3) pays off most here.

7 · LangChain: create_agent (built on LangGraph)

Our 20-line loop becomes ~6 lines. The framework owns the loop, the tool execution, the state, and the step guard.

In LangChain v1 the idiomatic agent is create_agent (which runs on LangGraph under the hood — the right tool for stateful, looping control flow, as we flagged in Lesson 5):

actual output
It looks like you were charged twice for the Plus monthly plan on June 1, 2026. I've escalated this to a human agent who will review the duplicate charge and contact you within 24 hours. Your current plan is Plus, and it renews on July 15, 2026.

Same result as the raw loop — it called both tools, found the duplicate charge and the plan, and escalated — but the loop, tool dispatch, and step guard are all handled for us.

✦ This is where the framework earns its keep

For a one-shot call, LangChain barely beat the raw SDK (Lesson 1). For an agent loop with state, tool dispatch, and guards, create_agent removes real, fiddly code. The trade you accept: less visibility into the loop and the framework's choices about how it runs. You've now built it both ways — so you can pick with eyes open.

8 · When NOT to use an agent

The most important judgment in this lesson. Agents are seductive and usually overkill.

An agent buys you dynamic step selection at the cost of predictability, latency, and money. If you already know the steps, you don't need it — a Lesson 5 pipeline is cheaper, faster, easier to test, and can't loop. Reach for an agent only when the steps genuinely vary per request and you can't enumerate them in advance.

Use a pipeline (L5) when…Use an agent (L7) when…
The steps are always the same.The needed steps differ per request.
You want predictable cost/latency.You can bound the worst case and accept variance.
Most support tickets (classify → answer)."Investigate my account and figure out X."
⚠ "Agentic" is not a goal

A surprising amount of "agent" work is better as a chain plus a couple of ifs. Start with the simplest thing that solves the task; promote to an agent only when fixed control flow truly can't express it. Simpler is cheaper, faster, and more reliable.

🔨 Try / modify this

  1. Watch the loop choose. Run run_agent on a question needing only one tool, then one needing both. The step count changes because the model decides.
  2. Force the guard. Set max_steps=1 and ask a two-lookup question. Watch it hit the cap and return the "step limit reached" message instead of looping.
  3. Trigger a tool error. Ask about nobody@example.com. The tool returns {"error": ...}; see the agent recover and ask for a valid email.
  4. Add a third tool. Give it get_plan_limits(plan) and ask a question spanning all three. How many steps does it take?
  5. Compare to a pipeline. Solve the same double-charge question with a fixed L5 chain. Which was simpler? For this task, the agent may be overkill — that's the lesson.

Few-shot prompting

Zero-shot gives you a plausible paragraph. But the first real job in our support system is routing — turning each incoming message into one machine-readable label. For that you need consistency, not eloquence. Few-shot prompting buys it for the price of a handful of examples.

1 · The job: routing tickets

A support agent's first decision is "what kind of message is this?" — so it can route to the right team, template, or pipeline branch.

We need every Nimbus message sorted into exactly one of five labels: billing, bug, how_to, cancellation, other. The output has to be one of those exact strings — because the next step in our code does if label == "billing": …. A nice paragraph is useless here.

Let's see what zero-shot (Lesson 1's technique) does with that constraint:

representative output
'This message is about billing.' 'Bug' 'how_to'

Three messages, three different output shapes: a full sentence, a capitalized word with no match to our label set, and one that happens to be right. The model understood the task fine — it just won't commit to a format. Your if label == "billing" check breaks on the first row, and "Bug" != "bug" breaks on the second.

✗ The core problem

Zero-shot can't read your mind about the output contract. You described the labels in words, but "describe in words" leaves room for interpretation. We need to show, not tell.

2 · What few-shot is, and why it works

Stop describing the task. Demonstrate it.

Few-shot prompting means putting a handful of worked examples (input → correct output) into the prompt before the real input. The model reads the pattern and continues it. That's it — no fine-tuning, no training, no weight changes. The learning happens entirely inside the prompt at inference time, which is why it's also called in-context learning.

Why it works

Language models are pattern-completers. When the prompt establishes a rhythm — message → label, message → label, message → — the most probable continuation is another label in the same shape. Examples do two jobs at once that a written instruction does poorly:

  • They pin the output format. Five examples that all answer with a bare lowercase label make "a bare lowercase label" overwhelmingly the likely next token.
  • They disambiguate the task. A borderline case ("upgrade me to Pro and bill me" → billing) teaches a boundary you'd struggle to phrase in a rule.

When to reach for it

  • You need a consistent, fixed output format (labels, tags, short structured text).
  • The task is easier to show than to describe (tone, style, edge cases).
  • Classification, extraction, formatting, or style-matching on a small label space.

3 · Build the router from scratch

We'll encode the examples as a fake conversation — alternating user / model turns — then append the real message. This is the cleanest form of few-shot, and it maps directly onto the LangChain version later.

representative output
billing <- My card got declined but I still see the charge. bug <- Sync keeps failing with error code 0x83. how_to <- Where's the button to invite a teammate? --- 3 calls | 396 in + 6 out tokens | ~$0.000042

Clean, lowercase, parseable labels — every time. Two details that matter:

  • temperature=0 makes it deterministic. For a classifier you want the same input to always produce the same label; creativity is the enemy here.
  • normalize() is your seatbelt. Even a great prompt occasionally returns something off-list; we coerce to a known label instead of trusting raw text. (Lesson 6 will make this guarantee structural with enums — but defensive parsing is always good hygiene.)
ℹ Notice the token count

396 input tokens for 3 calls = ~132 per call, versus ~15 for the zero-shot version. That extra ~117 tokens is the six examples, re-sent every single call. Few-shot trades input tokens for reliability. Hold that thought for the cost section.

4 · Does it actually help? A 60-second eval

"It looks better" is not evidence. Even a tiny labelled set turns a vibe into a number — this is the seed of the real evaluation we build in Lesson 9.

representative output
zero-shot accuracy: 67% few-shot accuracy: 100%

On this set, few-shot fixes the two cases zero-shot fumbled — the borderline "upgrade + bill" (correctly billing) and the formatting that broke parsing. The improvement is real and measurable, and now you can prove a prompt change helped instead of hoping.

⚠ Don't over-trust a 6-row eval

Six examples can't tell you the difference between 95% and 99% accuracy, and a lucky run can read as "100%". This is a smoke test, not a benchmark. Lesson 9 builds a proper eval set, handles non-determinism, and adds metrics beyond raw accuracy. For now: small evals beat no evals.

5 · Failure modes

Few-shot is reliable, not magic. The failures are specific and avoidable once you know them.

FailureWhat happensFix
Example imbalanceIf 4 of 6 examples are billing, the model over-predicts billing — it learns your distribution, not just the task.Balance examples across labels; cover each class at least once.
Order / recency biasThe label of the last example can tug ambiguous inputs toward it.Shuffle example order; don't end on the rarest class.
Format driftOccasional stray punctuation, casing, or an explanation despite the examples.temperature=0, a firm "ONLY the label" instruction, and always normalize().
Out-of-distribution inputA message unlike any example gets a confident but wrong label.Keep an explicit other class; later, let it abstain / escalate (Lessons 6–7).
Too many examplesPast ~5–10, accuracy plateaus while cost and latency keep climbing.Use the fewest examples that hit your accuracy target; measure (section 4).
✗ Subtle trap: label leakage

If your example inputs literally contain the label word ("This is a billing question…"), the model learns to keyword-match instead of reason. Write examples that look like real user messages — messy, indirect, label-word-free — or you'll measure the wrong skill.

6 · Cost, latency & the caching trick

Few-shot's one real cost: those example tokens ride along on every single request.

Because the API is stateless, your six examples (~120 tokens) are sent on every call. At Nimbus's imagined 50,000 tickets/day on gemini-2.5-flash-lite ($0.10 / 1M input):

Input tokens / callInput cost / day (50k calls)
Zero-shot~15~$0.075
Few-shot (6 examples)~135~$0.68

Pennies here — but the shape matters: the cost is the example prefix × call volume, and it grows every time you add an example. Two levers keep it down:

  • Use the cheapest model that passes your eval. Routing doesn't need flash, let alone pro. We measured (section 4) and flash-lite hit 100%.
  • Cache the static prefix. The examples never change between calls — the perfect case for context caching.
◆ Prompt / context caching

Gemini can reuse a repeated prompt prefix instead of re-charging full price for it. Implicit caching applies an automatic discount when consecutive requests share a long prefix; explicit caching (client.caches.create(...)) lets you register the example block once and reference it cheaply. For a high-volume classifier with a fixed example set, this is the standard production move. (Caches have a minimum token size and a TTL — worth it at scale, overkill for a demo.)

Latency: a longer prompt adds a little input-processing time, but the output is a 1–3 token label, so few-shot routing stays fast. Output length, not example count, dominates latency.

7 · The same thing in LangChain

LangChain has a purpose-built object for exactly this: FewShotChatMessagePromptTemplate.

Instead of hand-building the alternating user/model turns, you declare an example shape once and hand LangChain the list. The examples expand into chat turns for you, and the whole thing composes into an LCEL chain (prompt | llm | parser).

representative output
billing <- My card got declined but I still see the charge. bug <- Sync keeps failing with error code 0x83.

What did the framework do for us?

ConcernRaw SDKLangChain
Building example turnsWe loop and create Content objectsFewShotChatMessagePromptTemplate expands them
Templating the inputString building by hand{input} placeholder in a reusable prompt
Output handlingWe read resp.textStrOutputParser in the chain
Picking examplesStatic list (you choose)Static or a dynamic ExampleSelector
ℹ The feature worth knowing: dynamic example selection

With a large, diverse example pool, sending all of them is wasteful and can hurt. LangChain's SemanticSimilarityExampleSelector embeds your pool and, per request, picks the few examples most similar to the incoming message. That's few-shot + retrieval — and it's a direct preview of Lesson 8 (RAG), where we build embeddings and similarity search from scratch. For a fixed 6-example router, the static list is simpler and cheaper.

8 · Trade-offs: when to reach for few-shot

DimensionFew-shot
Cost🟠 Higher than zero-shot — examples are re-sent every call. Mitigate with a cheap model + context caching.
Latency💚 Still fast for short outputs; the extra input tokens add little.
Reliability💚 Big jump over zero-shot for format + edge-case consistency.
Reach for it when…You need a consistent format/label, the task is easier shown than described, and the label space is small.
Reach for something else when…You need a hard output guarantee (→ structured output / enums, Lesson 6), the example pool is huge (→ dynamic selection / RAG, Lesson 8), or the task needs multi-step reasoning (→ CoT, Lesson 4).

🛠 Try / modify this

  1. Break it on purpose. Delete the cancellation example, then classify "I'd like to terminate my plan." Watch it mislabel — proof that examples define the reachable label space.
  2. Find the floor. Drop from 6 examples to 2, then 1, re-running eval_router.py each time. How few can you keep before accuracy falls? That's your cost/reliability sweet spot.
  3. Test recency bias. Move the single other example to the very end of EXAMPLES and re-run on borderline inputs. Did anything drift toward other?
  4. Swap the model. Change flash-lite → flash and compare accuracy and llm.usage.report(). Is the pricier model worth it for this task? (Usually: no.)
  5. Add a label. Introduce feedback (feature requests/praise), add one example, extend the test set, and re-measure. This is how you grow a classifier safely.

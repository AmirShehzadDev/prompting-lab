Chain-of-thought & variants

The agent has a voice and a router, but some support questions need actual reasoning — apply a rule, do some math, weigh clues. Ask for the answer directly and the model often blurts a confident wrong number. Chain-of-thought makes it show its work first — and that single change can flip wrong to right.

1 · A question that needs reasoning

Real support isn't all lookups. Some tickets require applying a rule and doing math — exactly where models slip if you rush them.

Here's a self-contained task. We hand the model the facts it needs (a real agent would retrieve these — Lesson 8), then ask a question with a deliberate trap:

The correct chain: the 600 GB folder is owned by the colleague, so it doesn't count → your usage is 1,600 GB → that fits Plus (2,048 GB) → 6 × $10 = $60. The trap: add the 600 GB and you get 2,200 GB → "need Pro" → $150. Wrong.

Now ask for the answer directly — no room to reason (we also switch off the model's built-in thinking, explained in section 6, so we can see the classic behavior):

representative output — wrong
FINAL: plan=Pro, total=$150

It fell for the trap: it summed 1,600 + 600 and skipped the ownership rule. When you force a one-token-style jump straight to the answer, the model has nowhere to do the multi-step work — so it pattern-matches the most salient numbers and gets it wrong.

2 · Zero-shot CoT: "think step by step"

The whole technique is one instruction: ask the model to reason before it answers. No examples required — hence "zero-shot" CoT.

representative output — correct
Step 1 — Apply the quota rule. The 600 GB team folder is owned by my colleague, so as a member-only it does NOT count against my quota. Step 2 — My usage = my own files only = 1,600 GB. Step 3 — Cheapest plan that fits 1,600 GB: Free (15 GB) is too small; Plus (2,048 GB) fits. So Plus. Step 4 — Cost = 6 months x $10/mo = $60. FINAL: plan=Plus, total=$60

Why this works

Generating reasoning gives the model scratch space. Each token it writes becomes part of the context for the next token, so "the folder is owned by someone else → it doesn't count" is now on the page when it computes the total. You're letting it spend more computation, laid out in sequence, instead of forcing a single intuitive leap. For any task with multiple dependent steps — math, rule application, multi-clue diagnosis — this is the highest-value prompt change you can make.

ℹ It's the reasoning, not the magic words

"Let's think step by step" is just a convenient trigger. What matters is that the model produces intermediate steps before the answer. "Show your work", "reason through it first", or naming the sub-steps (as we did: "apply the rule, then do the math") all work — often better, because they're specific to your task.

3 · Getting just the answer back out

Reasoning is great for accuracy, but downstream code (and your users) want the answer, not the monologue.

That's why we made the model end with a machine-readable FINAL: line. Parse it, and throw the prose away:

⚠ Don't show raw chain-of-thought to users

The reasoning trace is verbose, costs output tokens, can expose internal logic, and sometimes looks wrong even when the final answer is right. Treat it as scratch work: generate it to get accuracy, parse out the FINAL line, and surface only that. (Lesson 6 makes this extraction bulletproof with structured output instead of regex.)

4 · Few-shot CoT: teaching the reasoning style

When zero-shot CoT reasons in a shape you don't want — wrong order, wrong format, missing a domain rule — show it how, by combining Lesson 2's examples with reasoning.

Each example now includes the worked-out steps, not just the answer. The model copies the method, not only the format:

Example 1 demonstrates the "member → doesn't count" path; Example 2 the "owner → counts" path. Together they teach the exact rule our trap depends on. Use few-shot CoT when zero-shot CoT is close but inconsistent; it costs more tokens (the examples ride every call, per Lesson 2), so reach for it only when the cheaper zero-shot version doesn't clear your eval.

5 · Self-consistency: sample & vote

Even with CoT, a single run can take a wrong turn. Self-consistency hedges: reason several times with some randomness, then take the majority answer.

The idea: a correct answer tends to be reachable by many reasoning paths, while mistakes scatter. So sample N chains at temperature > 0 and vote on the parsed result.

representative output
{'answer': ('plus', 60.0), 'votes': 4, 'of': 5} 5 calls | 1985 in + 1240 out tokens | ~$0.0037

Four of five paths landed on Plus/$60; one slipped to Pro/$150. The vote absorbs that one bad run. The cost is right there in the report: 5× the calls. Self-consistency buys reliability with money and latency — use it for high-stakes reasoning, not for every request.

✦ Voting needs a comparable answer

Self-consistency only works when you can extract a normalized answer to vote on — which is exactly why we parse a structured FINAL line. You can't majority-vote freeform paragraphs. Reasoning varies; the answer must be comparable.

6 · The modern twist: native "thinking" models

Here's the part the old CoT tutorials don't mention: recent models reason internally by default. We've been switching that off to teach the classic technique.

Gemini 2.5 models have built-in thinking — they can generate hidden reasoning before the visible answer. That's why we passed thinking_budget=0 in every demo above: to isolate prompted CoT. Turn thinking back on, and the model often nails the trap while you ask for only the final line — the reasoning just happens out of sight:

representative output — correct, no visible reasoning
FINAL: plan=Plus, total=$60

Same correct answer as prompted CoT, but clean output. So is explicit chain-of-thought obsolete? No — it's now one of two levers, and you should know both:

Use…When
Native thinking (thinking_config)You're on a thinking-capable model and want clean output with the reasoning hidden; simplest path.
Prompted CoTYou need the reasoning visible (debugging, audit), you're on a non-thinking/cheaper model, or you want to control the exact reasoning format.
Self-consistencyHigh-stakes accuracy — wrap either of the above and vote.
◆ Thinking isn't free

Hidden reasoning tokens are billed as output (they show up as usage_metadata.thoughts_token_count) and add latency. thinking_budget is your dial: higher for hard problems, 0 for simple ones where thinking is wasted effort. It's the same cost/accuracy trade-off as prompted CoT — just managed by the model instead of your prompt.

7 · Failure modes

Reasoning helps, but it introduces its own ways to go wrong.

FailureWhat happensMitigation
Confident rationalizationThe model writes fluent reasoning that justifies a wrong answer.Self-consistency; verify the answer against ground truth where possible (Lesson 9).
Unfaithful reasoningThe written steps aren't actually why it answered — CoT is not a true explanation.Don't treat the trace as an audit log; trust the answer's correctness, not the prose.
OverkillCoT on a trivial task (e.g. routing) just adds cost, latency, and sometimes errors from overthinking.Reserve CoT for genuinely multi-step problems; keep classification one-shot.
Systematic biasIf the model is reliably wrong, the majority vote is reliably wrong too.Self-consistency fixes variance, not bias — fix the prompt/facts.
Format / truncationThe FINAL line is missing, or long reasoning hits max_output_tokens.Generous token cap; parse_final returns None → retry or fall back.

8 · Cost, latency & reliability

Reasoning is paid for in output tokens — the most expensive kind.

  • Cost: CoT generates a reasoning trace (or hidden thinking tokens), all billed at the output rate (~8× input on flash). A direct answer was ~6 output tokens; the CoT trace was ~250. Self-consistency multiplies that by N.
  • Latency: output is generated token-by-token, so more reasoning = proportionally slower. Self-consistency's N calls can run in parallel to hide wall-clock latency, but not cost.
  • Reliability: a real lift on multi-step problems; neutral-to-negative on simple ones. The discipline is knowing which is which — measure (Lesson 9), don't assume.
✦ Rule of thumb

Direct answer for simple tasks → CoT (or thinking) for multi-step reasoning → add self-consistency only when an error is expensive and you can afford N× the calls. Escalate deliberately; each step up costs more.

10 · Trade-offs & try / modify

VariantCostReliabilityReach for it when…
Direct answer💚 Lowest🟠 Poor on multi-stepSimple, single-step tasks.
Zero-shot CoT🟠 +reasoning tokens💚 Strong on multi-stepMulti-step reasoning; the default upgrade.
Few-shot CoT🟠 +examples +reasoning💚 Strong, consistent styleZero-shot CoT is close but reasons the wrong way.

🛠 Try / modify this

  1. Watch the trap spring. Run answer_direct() a few times, then answer_cot(). See direct slip to Pro/$150 while CoT holds Plus/$60.
  2. Find the thinking budget that works. Call answer_thinking(budget=0), then 256, then 2048. Where does it start getting the trap right? That's your accuracy/cost dial.
  3. Make self-consistency disagree. Raise temperature to 1.5 and run self_consistent(). More split votes — see reliability degrade as randomness rises.
  4. Break the parser. Change the prompt's final-line format and watch parse_final return None. This is the fragility Lesson 6 removes.
  5. Prove "overkill is real". Apply CoT to the Lesson 2 routing task and compare accuracy + cost vs the plain few-shot router. Usually: same accuracy, more cost.
✦ What you built

A reasoning step for Nimbus — zero-shot & few-shot CoT, a structured FINAL answer you can parse, self-consistency voting, and a working grasp of native thinking — plus the judgment to know when reasoning earns its cost and when it's just burning tokens.

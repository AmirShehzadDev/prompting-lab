Prompt chaining & self-refine

We've built the pieces — a router (L2), a persona (L3), reasoning (L4). Now we stop cramming everything into one call and wire them into a pipeline: classify → route → draft → critique → refine. This is the real shape of a support agent, and where the framework finally earns its keep.

1 · One mega-prompt vs. a chain

You could ask one giant prompt to "classify, then route, then write a perfect reply, then check it." You shouldn't.

A single prompt juggling many instructions is the brittle option: it's hard to debug (which instruction did it drop?), you can't use a cheap model for the easy parts, and you get no inspection points between steps. Prompt chaining breaks the task into a sequence of focused calls, each doing one thing and passing its output forward.

classifyL2 router→ routepick handling→ draftL3 persona→ critiquerubric check⟲ refinefix & repeat

Each stage is simpler and more reliable than the whole, individually testable, and free to use a different model (cheap flash-lite to classify, capable flash to draft). The trade-off — more calls, more latency, more cost — is the rest of this lesson.

✦ Chaining reuses everything you've built

This pipeline doesn't invent new prompts — it orchestrates the router from Lesson 2 and the persona from Lesson 3 into stages. That's the point of building one coherent system: the parts compose.

2 · Build the pipeline: classify → route → draft

Three stages. The first is literally the Lesson 2 router, imported. The second is plain Python. The third reuses the Lesson 3 persona.

representative output
billing I'm sorry about the double charge — I completely understand the frustration. I'm not able to issue refunds myself, but I can get this sorted: could you confirm the date of the duplicate $10 charge? I'll summarize it and escalate to a human billing agent who can review and process any refund. Thanks for your patience.

Notice what each stage contributed: classify picked billing, route supplied the "don't promise refunds, escalate" guidance, and draft rendered it in the Nimbus voice. Three small, debuggable steps instead of one sprawling prompt. But is this draft actually good? Let's not assume — let's check it.

3 · Self-refine: critique → improve

A second model pass critiques the draft against a rubric, then a third rewrites it. Critiquing is easier than generating, so a focused critic catches what the drafter missed.

The critic returns a machine-checkable verdict — PASS or FAIL: … — which is what lets us drive a loop in code. Run it on a weaker draft:

critique() on a draft that forgot to ask for the charge date
FAIL: Doesn't ask for the specific charge date/details needed to escalate the billing issue.
✦ Why a separate critic beats "just write it well"

Asking one call to both write and self-check splits its attention. A dedicated critique pass starts fresh, reads the draft as an outsider, and judges against an explicit rubric — so it catches the missing next-step that the drafter, focused on tone, glossed over. Generation and evaluation are different skills; give each its own call.

4 · Putting it together: handle()

Now the full chain, with the critique→refine loop bounded by a hard cap. Always cap the loop — never let it run "until perfect".

representative run
[bug] refines=1 (passed) I'm sorry sync keeps stopping — let's fix it. Try these in order: 1. Confirm you're online and signed in. 2. Quit Nimbus fully and reopen it, then click Resume on the sync status. 3. Check you still have free storage space. If it's still stopping, send me your operating system, your Nimbus app version, and the exact error text, and I'll take it from there. --- 2 calls | 520 in + 240 out tokens | ~$0.00076 (drafter) 2 calls | 900 in + 40 out tokens | ~$0.00037 (critic)

Round 1's draft skipped asking for the version/error; the critic caught it, and the refine pass added it — then round 2 passed. The reply is the product of five calls across two models (one cheap classify + two drafts + two critiques), each doing one job well.

5 · Failure modes

Chains multiply power — and multiply the ways things break.

FailureWhat happensMitigation
Cascading errorsA wrong classify routes everything downstream wrong — the polished reply answers the wrong question.Validate between stages; the critic can catch off-category replies; fall back to a safe generic handler on low confidence.
Latency stacking5 sequential calls = sum of 5 latencies; the user waits.Parallelize independent stages; use fast models for simple ones; cap refine rounds.
Cost multiplicationOne ticket now costs 5 calls instead of 1.Cheapest model per stage; skip refine when the first draft passes; cache static prefixes.
Refine oscillationThe loop edits forever, gold-plates, or makes the reply worse chasing the rubric.Hard max_rounds cap; stop on PASS; keep the best version, not the last.
Mis-calibrated criticCritic rubber-stamps everything (always PASS) or never passes.Concrete, testable rubric; evaluate the critic itself (Lesson 9).
⚠ The cascade is the big one

In a chain, every stage trusts the one before it. A 90%-accurate classifier means 10% of tickets are confidently mishandled end-to-end, no matter how good the later stages are. Reliability is multiplicative — measure each stage (Lesson 9), and put cheap validation between them.

6 · Cost, latency & reliability

Chaining trades calls for control. Know the exchange rate.

  • Cost: a chain costs the sum of its stages. Our pipeline is 1 cheap classify + up to 2 drafts + up to 2 critiques. Keep it down by using the cheapest model per stage and skipping refine when the first draft passes.
  • Latency: sequential stages add up — the user waits for all of them. Stages with no data dependency can run in parallel (e.g., classify and retrieval in Lesson 8). The refine loop is inherently sequential, so cap it.
  • Reliability: up, via inspection points, a dedicated critic, and small testable steps — but watch the cascade. A chain is only as reliable as its weakest stage times all the rest.

7 · LangChain: LCEL (and where you outgrow it)

This is what LCEL — the | pipe composition — is for: wiring a linear chain of steps into one runnable object.

The linear spine — classify → attach guidance → draft — is three composable Runnables joined with |. That's LCEL at its best: declarative, streamable, and the same .invoke() interface everywhere.

ℹ "Is LCEL the latest way?" — yes, for chains

In LangChain v1, LCEL (the | pipe / Runnable / .invoke()) is current and recommended for linear composition — it was not deprecated. What got retired into the langchain-classic package are the old class-based chains (LLMChain, SequentialChain). The rule of thumb in v1:

LCEL for linear chains → create_agent for agents → LangGraph for stateful / branching / looping orchestration.

⚠ Where LCEL stops: the refine loop

Notice we only built the linear part in LCEL. The critique → refine loop is a conditional cycle ("if FAIL, go back; else stop") — and cyclic, stateful control flow is exactly what LCEL doesn't express cleanly. That's the signal to reach for LangGraph, which models nodes, conditional edges, and loops as a graph with state. We build the agent loop properly with LangGraph in Lesson 7; here, a plain Python for loop around the LCEL chain is the honest, simple choice.

8 · Trade-offs & try / modify

ApproachCost / latencyReliabilityReach for it when…
One mega-prompt💚 1 call🟠 Brittle, hard to debugSimple, single-step tasks.
Linear chain🟠 Sum of stages💚 Modular, testable, inspectableMulti-step tasks with clear stages (this pipeline).
+ self-refine🔴 +2 calls/round💚💚 Higher quality outputQuality matters and you can afford the loop. Always cap it.

🛠 Try / modify this

  1. Watch the loop work. Run handle() on a vague message and print the draft before and after refine. What did the critic catch?
  2. Cause a cascade. Hard-code intent = "how_to" for a billing message and watch a fluent, well-refined — but wrong-category — reply come out. That's why stage 1 accuracy dominates.
  3. Tune the cap. Set max_rounds=0 (no refine), then 1, then 3. Where does quality stop improving? Each extra round is two more calls.
  4. Tighten the rubric. Add a 5th check ("must not exceed 90 words") and watch refine fight to satisfy it — and occasionally over-trim. That's refine oscillation.
  5. Compare raw vs LCEL. Run pipeline.py and pipeline_lc.py on the same message. Same result — decide which you'd rather maintain.

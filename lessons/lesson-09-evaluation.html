Evaluation & optimization

We've spent eight lessons claiming each technique made things "better". Says who? This lesson replaces vibes with numbers: a golden set, three kinds of metric (exact-match, LLM-as-judge, groundedness), and a way to prove a prompt change actually helped instead of quietly breaking something else.

1 · Why "better" needs a number

Prompt tuning without measurement is a game of whack-a-mole in the dark.

You tweak the system prompt to fix one rude reply, ship it, and silently break the refund-escalation behavior you got right in Lesson 3. Without a way to measure, you'd never know. Eval turns prompt engineering from guesswork into something you can regression-test like any other code:

  • A golden set — representative inputs with known-good answers/labels.
  • A metric — a function that scores an output against the gold.
  • A runner — applies your system to the set and reports the score.

The hard part is the metric, because LLM outputs range from objective (a label is right or wrong) to subjective (is this reply good?). So we use three kinds, matched to the task:

MetricForHow
Exact-match / accuracyObjective tasks (the router's labels)Compare to gold; count hits.
LLM-as-judgeSubjective quality (the support replies)A model scores the output against a rubric.
Groundedness / faithfulnessRAG answersCheck the answer is supported by the retrieved context.

2 · The golden set

Small, representative, human-labeled, version-controlled. It grows from your real failures.

Ours lives in data/evalset.jsonl — one labeled example per line. Start tiny; every time a real bug slips through, add it as a new row so it can never regress unnoticed:

✦ The eval set is a product asset

It's the single most valuable artifact for shipping LLM features confidently. Treat it like code: commit it, review additions, and grow it deliberately from real misses. A 10-row set catches gross regressions; a few hundred well-chosen rows is a serious quality gate.

3 · Metric 1: exact-match accuracy

For objective tasks like the router, the metric is simple: did the predicted label equal the gold label?

representative output
few-shot router accuracy: 90%

Now "the router is good" is a number you can track. If a prompt change drops it to 80%, your CI catches the regression before your customers do. Exact-match is cheap, deterministic, and the gold standard when a single right answer exists.

4 · Metric 2: LLM-as-judge

Most support replies have no single right answer. To score quality at scale, use a model as the grader — with a rubric and structured output (Lesson 6) so the verdict is machine-readable.

representative output
5 True - Directly confirms the double charge with specifics and gives a clear next step (escalation).
⚠ The judge is a model — treat it like one

LLM judges have biases: they favor longer answers, their own style, and the first option in a pairwise test. Mitigate by: a concrete rubric (not "is it good?"), temperature=0, validating the judge against a sample of human labels, and trusting it most for relative comparisons (A vs B) rather than absolute scores. A judge calibrated against humans is invaluable; an uncalibrated one is just a confident opinion.

5 · Metric 3: groundedness (RAG faithfulness)

For RAG (Lesson 8), the key question isn't "is it good prose?" but "is every claim actually supported by the retrieved context?"

representative output
supported=True unsupported_claims=[]

If unsupported_claims is non-empty, your RAG system is hallucinating beyond its sources — a faithfulness failure you can now catch automatically, even when the answer sounds perfect.

6 · Evaluating non-deterministic output

The same prompt can score differently on two runs. Your eval has to account for that.

  • Pin temperature=0 for both the system under test and the judge. It won't be perfectly deterministic, but it minimizes run-to-run variance.
  • Sample and aggregate. For anything stochastic, run each case a few times and report the pass rate (e.g., "passes 9/10 runs"), not a single lucky/unlucky result.
  • Report variance, not just the mean. A prompt that scores 85% ±2% is more shippable than one that scores 88% ±15%.
ℹ A 1-point eval move on 10 rows is noise

With a tiny set, small score differences are within the noise band. Either grow the set or run multiple trials before declaring a winner — otherwise you'll "optimize" against randomness.

7 · Optimization: prove a change helped

With a metric in place, optimization is a loop: change one thing, re-measure, keep it only if the number went up (and nothing else went down).

The simplest experiment is an A/B on a single change. Remember Lesson 2's claim that few-shot beats zero-shot? Now we prove it on the eval set instead of asserting it:

representative output
zero-shot router: 70% few-shot router : 90%

That's the whole discipline: every prompt decision in this course could be settled this way instead of by intuition. Wrap it in a loop — try a tweak, run the suite, keep or revert — and you're doing prompt optimization.

✦ DSPy: let the optimizer write the prompt

You can automate the loop. DSPy takes a different stance entirely: you declare a signature (inputs → outputs) and a metric, and its optimizer searches over instructions and few-shot examples to compile a prompt that maximizes your metric — no hand-tuning. It's the logical end of this lesson: if you have a metric, a machine can optimize the prompt against it. (Different library from our stack; worth knowing it exists when hand-tuning stalls.)

8 · Frameworks: structured judges, LangSmith, DSPy

Eval is mostly your own harness regardless of stack — but frameworks help with the judge, the datasets, and the optimization.

The LLM-as-judge is cleaner with with_structured_output (Lesson 6), which guarantees the verdict shape:

  • LangSmith (LangChain's platform) provides hosted datasets and an evaluate() runner with off-the-shelf evaluators (exact-match, embedding distance, LLM-as-judge) and a UI to inspect results — the productionized version of the harness above.
  • DSPy goes further and optimizes the prompt against your metric (§7).

For learning, rolling your own harness (as we did) is the right move: you see exactly what's measured. Reach for the platforms when you need scale, collaboration, and dashboards.

🔨 Try / modify this

  1. Run the A/B. Execute evaluate.py and confirm few-shot beats zero-shot on the eval set — the Lesson 2 claim, now measured.
  2. Grow the set from a failure. Find a message the router mislabels, add it to evalset.jsonl, and watch accuracy drop. Fix the prompt until it passes again.
  3. Calibrate the judge. Hand-score 5 replies yourself, then compare to judge(). Where do you disagree? Tighten the rubric.
  4. Catch a hallucination. Feed check_grounded an answer with a made-up claim and confirm it lands in unsupported_claims.
  5. Measure variance. Run the judge on the same reply 5 times at temperature=0 vs 0.8. How stable is each?
